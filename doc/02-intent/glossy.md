# AIOps 拓扑管理系统 - 术语表

## 核心概念

### 拓扑相关

- **拓扑图（Topology Graph）**：展示系统中各种资源及其相互关系的有向图或无向图，用于可视化和分析系统架构。

- **节点（Node）**：拓扑图中的基本单元，代表一个具体的 IT 资源实体（如应用、中间件、数据库等）。

- **边（Edge）**：连接两个节点的关系线，表示节点之间的依赖、调用、数据流等关系。

- **拓扑关系（Topology Relationship）**：节点之间的关联关系，包括依赖关系、调用关系、部署关系等。

- **拓扑层级（Topology Layer）**：拓扑图的分层结构，如基础设施层、中间件层、应用层、接口层等。

### 节点类型

- **应用节点（Application Node）**：代表一个应用系统或服务的节点，是业务功能的载体。

- **中间件节点（Middleware Node）**：代表中间件软件的节点，如消息队列（Kafka、RabbitMQ）、缓存（Redis）、服务网格等。

- **数据库节点（Database Node）**：代表数据库实例的节点，如 MySQL、PostgreSQL、MongoDB 等。

- **代码仓库节点（Repository Node）**：代表代码仓库的节点，如 Git 仓库、SVN 仓库等。

- **接口节点（Interface Node）**：代表 API 接口或服务端点的节点，包括 REST API、gRPC、WebSocket 等。

- **报表节点（Report Node）**：代表数据报表或数据看板的节点。

- **主机节点（Host Node）**：代表物理机或虚拟机的节点。

- **容器节点（Container Node）**：代表 Docker 容器或 Kubernetes Pod 的节点。

- **网络设备节点（Network Device Node）**：代表网络设备的节点，如负载均衡器、网关、交换机等。

### 关系类型

- **依赖关系（Dependency）**：表示一个节点依赖另一个节点才能正常运行，如应用依赖数据库。

- **调用关系（Invocation）**：表示一个节点主动调用另一个节点的服务或接口。

- **部署关系（Deployment）**：表示一个节点部署在另一个节点上，如应用部署在主机上。

- **数据流关系（Data Flow）**：表示数据在节点之间的流动方向和路径。

- **归属关系（Ownership）**：表示节点的所有权或管理权归属，如应用归属于某个团队。

- **关联关系（Association）**：表示节点之间的一般性关联，如代码仓库关联到应用。

### 节点属性

- **节点标识（Node ID）**：节点的唯一标识符。

- **节点名称（Node Name）**：节点的显示名称。

- **节点类型（Node Type）**：节点所属的类型分类。

- **节点状态（Node Status）**：节点的运行状态，如运行中、停止、异常、未知等。

- **节点元数据（Node Metadata）**：节点的附加属性信息，如版本号、配置参数、标签等。

- **节点标签（Node Label/Tag）**：用于分类和筛选节点的标记，如环境标签（生产、测试）、业务标签等。

## 管理功能

### 节点管理

- **节点注册（Node Registration）**：将新的 IT 资源注册为拓扑图中的节点。

- **节点发现（Node Discovery）**：自动发现和识别系统中的 IT 资源并创建对应节点。

- **节点更新（Node Update）**：修改节点的属性信息或状态。

- **节点删除（Node Deletion）**：从拓扑图中移除节点。

- **节点查询（Node Query）**：根据条件检索和查找节点。

### 关系管理

- **关系建立（Relationship Creation）**：在两个节点之间创建关系边。

- **关系更新（Relationship Update）**：修改关系的属性或类型。

- **关系删除（Relationship Deletion）**：移除节点之间的关系。

- **关系查询（Relationship Query）**：查询节点之间的关系路径。

### 拓扑分析

- **影响分析（Impact Analysis）**：分析某个节点故障或变更对其他节点的影响范围。

- **依赖分析（Dependency Analysis）**：分析节点的依赖链路和依赖深度。

- **路径分析（Path Analysis）**：分析两个节点之间的连接路径。

- **拓扑视图（Topology View）**：拓扑图的可视化展示方式，可按层级、业务域、环境等维度展示。

## 技术概念

### 图数据库

- **图数据库（Graph Database）**：专门用于存储和查询图结构数据的数据库，如 Neo4j、JanusGraph 等。

- **图查询语言（Graph Query Language）**：用于查询图数据的语言，如 Cypher、Gremlin 等。

- **图遍历（Graph Traversal）**：在图中按照特定规则访问节点和边的过程。

### 数据采集

- **数据源（Data Source）**：提供节点和关系数据的来源，如 CMDB、监控系统、配置中心等。

- **数据采集器（Data Collector）**：从数据源采集数据的组件。

- **数据同步（Data Synchronization）**：保持拓扑数据与实际系统状态一致的过程。

### 可观测性

- **监控指标（Metrics）**：节点的性能和健康状态指标，如 CPU、内存、响应时间等。

- **告警（Alert）**：节点或关系异常时触发的通知。

- **事件（Event）**：节点或关系状态变化的记录。

## 业务概念

### 组织结构

- **业务域（Business Domain）**：按业务功能划分的领域，如订单域、支付域、用户域等。

- **团队（Team）**：负责管理和维护节点的团队或组织。

- **环境（Environment）**：节点所属的运行环境，如开发环境、测试环境、生产环境等。

### 配置管理

- **配置项（Configuration Item, CI）**：CMDB 中的配置项，可以映射为拓扑图中的节点。

- **配置管理数据库（CMDB）**：存储 IT 资源配置信息的数据库。

- **变更记录（Change Record）**：节点或关系的变更历史记录。

## 扩展概念

### 智能运维

- **根因分析（Root Cause Analysis）**：通过拓扑关系分析故障的根本原因。

- **异常检测（Anomaly Detection）**：识别节点或关系的异常行为。

- **容量规划（Capacity Planning）**：基于拓扑关系进行资源容量规划。

### 可视化

- **布局算法（Layout Algorithm）**：拓扑图的自动布局算法，如力导向布局、层次布局等。

- **节点图标（Node Icon）**：节点在拓扑图中的可视化图标。

- **关系样式（Edge Style）**：边在拓扑图中的可视化样式，如实线、虚线、箭头等。
