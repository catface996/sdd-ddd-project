# 需求分析阶段评分标准

## 评分总则

- **评分制度**：扣分制（满分100分）
- **评分时机**：需求文档完成自检后
- **评分原则**：从完美状态开始，发现问题扣分

---

## 评分维度（100分）

### 1. 完整性（30分）

完整性评估需求文档是否全面覆盖了原始意图，包括功能需求和非功能性需求。这是需求质量的基础，不完整的需求会导致后续设计和实现的遗漏。

#### 1.1 功能需求覆盖（15分）

**评分项说明**：
- **遗漏核心功能**：-5分/项
- **遗漏次要功能**：-2分/项

**详细说明**：

**核心功能 vs 次要功能的判断标准**：
- **核心功能**：系统必须具备的功能，缺少则无法满足基本业务目标
  - 示例：电商系统的下单、支付功能
  - 影响：遗漏核心功能会导致系统无法使用或业务目标无法达成
  
- **次要功能**：辅助性功能，缺少不影响核心业务流程
  - 示例：电商系统的商品收藏、浏览历史功能
  - 影响：遗漏次要功能会降低用户体验，但不影响核心业务

**如何识别遗漏**：
1. **逐项对照原始需求**：将需求文档与原始需求文档逐项对照
2. **场景覆盖检查**：列出所有用户场景，检查是否都有对应的需求
3. **流程完整性检查**：检查业务流程是否完整（开始→处理→结束）
4. **边界情况检查**：是否考虑了异常场景、错误处理、边界条件

**常见遗漏类型**：
- 异常处理需求（如网络超时、数据校验失败）
- 边界条件需求（如空数据、超大数据量）
- 状态转换需求（如订单状态流转）
- 权限控制需求（如不同角色的操作权限）

**评分示例**：

**示例 1：遗漏核心功能**
- **原始需求**：用户可以创建、编辑、删除、查询项目
- **需求文档**：只包含创建和查询项目的需求
- **扣分**：遗漏编辑功能（-5分）、遗漏删除功能（-5分）
- **理由**：CRUD 是核心功能，缺少任何一项都会影响基本使用

**示例 2：遗漏次要功能**
- **原始需求**：用户可以导出项目列表为 Excel
- **需求文档**：未包含导出功能
- **扣分**：-2分
- **理由**：导出是辅助功能，不影响核心业务流程

**示例 3：遗漏异常场景**
- **原始需求**：用户登录系统
- **需求文档**：只描述了成功登录的场景
- **扣分**：遗漏密码错误场景（-2分）、遗漏账号锁定场景（-2分）
- **理由**：异常场景是必要的，但不是核心功能

**检查要点**：
- [ ] 是否覆盖了原始需求中的所有核心功能？
- [ ] 是否覆盖了所有用户场景？
- [ ] 是否考虑了边界情况和异常场景？
- [ ] 业务流程是否完整（开始→处理→结束）？
- [ ] 是否考虑了不同角色的功能需求？

---

#### 1.2 非功能性需求覆盖（15分）

**评分项说明**：
- **未提及性能要求**：-5分
- **未提及安全要求**：-5分
- **未提及可维护性**：-3分
- **未提及可扩展性**：-2分

**详细说明**：

非功能性需求（NFR）决定了系统的质量属性，是系统能否在生产环境稳定运行的关键。很多项目失败不是因为功能不完整，而是因为忽略了非功能性需求。

**1. 性能要求（-5分）**

**为什么重要**：
- 性能问题在生产环境才暴露，修复成本极高
- 影响用户体验和系统可用性
- 可能导致系统崩溃或无法使用

**应该包含的内容**：
- **响应时间**：API 响应时间、页面加载时间
  - 示例：API 响应时间 < 200ms（P95）
- **吞吐量**：系统能处理的并发请求数
  - 示例：支持 1000 QPS
- **资源使用**：CPU、内存、磁盘使用限制
  - 示例：单实例内存使用 < 2GB

**常见错误**：
- ❌ 使用模糊词汇："系统应该快速响应"
- ✅ 使用具体指标："API 响应时间 < 200ms（P95）"

**评分示例**：
- **完全未提及性能要求**：-5分
- **提及但不具体**（如"系统应该快"）：-3分
- **提及但不完整**（只有响应时间，没有吞吐量）：-2分

---

**2. 安全要求（-5分）**

**为什么重要**：
- 安全漏洞可能导致数据泄露、系统被攻击
- 合规要求（GDPR、等保等）
- 影响企业声誉和法律责任

**应该包含的内容**：
- **身份认证**：如何验证用户身份
  - 示例：支持 JWT Token 认证，Token 有效期 2 小时
- **授权控制**：如何控制用户权限
  - 示例：基于 RBAC 的权限控制，支持角色和资源级别的权限
- **数据保护**：敏感数据如何保护
  - 示例：密码使用 BCrypt 加密，敏感数据传输使用 HTTPS
- **审计日志**：记录关键操作
  - 示例：记录所有数据修改操作，包括操作人、时间、内容

**常见遗漏**：
- 未考虑 SQL 注入、XSS 等常见攻击
- 未考虑敏感数据的加密存储
- 未考虑 API 的访问控制
- 未考虑审计日志

**评分示例**：
- **完全未提及安全要求**：-5分
- **只提及认证，未提及授权和数据保护**：-3分
- **提及但不具体**（如"系统应该安全"）：-4分

---

**3. 可维护性（-3分）**

**为什么重要**：
- 系统上线后需要持续维护和问题排查
- 影响故障恢复时间（MTTR）
- 影响开发效率和运维成本

**应该包含的内容**：
- **日志记录**：记录什么级别的日志，日志格式
  - 示例：记录所有 API 请求和响应，包含请求 ID、耗时、状态码
- **监控指标**：需要监控哪些指标
  - 示例：监控 API 响应时间、错误率、系统资源使用率
- **错误处理**：如何处理和报告错误
  - 示例：统一的异常处理机制，返回标准错误格式
- **配置管理**：如何管理配置，是否支持动态配置
  - 示例：支持多环境配置，配置变更无需重启

**常见遗漏**：
- 未考虑日志记录策略
- 未考虑监控和告警
- 未考虑错误处理和用户友好的错误提示
- 未考虑配置管理

**评分示例**：
- **完全未提及可维护性**：-3分
- **只提及日志，未提及监控和错误处理**：-2分
- **提及但不具体**：-1分

---

**4. 可扩展性（-2分）**

**为什么重要**：
- 业务增长需要系统能够扩展
- 避免后期大规模重构
- 降低技术债务

**应该包含的内容**：
- **水平扩展能力**：是否支持增加实例来提升性能
  - 示例：系统设计为无状态，支持水平扩展
- **模块化设计**：是否易于添加新功能
  - 示例：采用 DDD 分层架构，模块间低耦合
- **数据扩展**：数据量增长时如何应对
  - 示例：预留分库分表能力
- **集成扩展**：是否易于集成第三方服务
  - 示例：采用适配器模式，易于替换第三方服务

**常见遗漏**：
- 未考虑未来的功能扩展
- 未考虑数据量增长
- 未考虑第三方服务的替换能力

**评分示例**：
- **完全未提及可扩展性**：-2分
- **提及但不具体**：-1分

---

**检查要点**：
- [ ] 是否明确了性能指标（响应时间、吞吐量等）？
- [ ] 是否考虑了安全需求（认证、授权、数据保护）？
- [ ] 是否考虑了可维护性（日志、监控、错误处理）？
- [ ] 是否考虑了可扩展性（未来扩展点）？
- [ ] 非功能性需求是否具体、可测量？

---

### 2. 准确性（25分）

准确性评估需求文档是否正确理解了原始意图，表述是否清晰无歧义。准确性问题会导致设计和实现偏离用户期望，造成大量返工。

#### 2.1 需求理解准确度（15分）

**评分项说明**：
- **严重理解偏差**：-10分/项
- **轻微理解偏差**：-5分/项

**详细说明**：

**严重理解偏差 vs 轻微理解偏差的判断标准**：

**严重理解偏差（-10分/项）**：
- **定义**：完全曲解了需求意图，导致后续设计方向错误
- **影响**：如果按此需求实现，将无法满足用户期望，需要大规模返工
- **特征**：
  - 理解的功能与原始意图完全相反
  - 遗漏了需求的核心目标
  - 添加了不必要的复杂功能

**示例 1：完全曲解需求**
- **原始需求**：用户可以查看自己创建的项目列表
- **需求文档**：用户可以查看系统中所有项目列表
- **问题**：理解错误，涉及权限和数据范围的根本性错误
- **扣分**：-10分
- **影响**：如果按此实现，会导致数据泄露和权限问题

**示例 2：遗漏核心目标**
- **原始需求**：系统需要支持多租户，不同租户的数据完全隔离
- **需求文档**：系统支持多用户，用户之间可以共享数据
- **问题**：完全遗漏了"数据隔离"这个核心目标
- **扣分**：-10分
- **影响**：架构设计会完全错误

**示例 3：添加不必要的复杂功能**
- **原始需求**：用户可以上传文件
- **需求文档**：系统需要支持文件版本管理、文件协作编辑、文件权限控制
- **问题**：过度解读，添加了原始需求未提及的复杂功能
- **扣分**：-10分
- **影响**：大幅增加开发成本和复杂度

---

**轻微理解偏差（-5分/项）**：
- **定义**：部分理解不准确，但不影响整体方向
- **影响**：需要局部调整，但不需要大规模返工
- **特征**：
  - 对细节的理解有偏差
  - 对优先级的判断有偏差
  - 对范围的理解有偏差

**示例 1：细节理解偏差**
- **原始需求**：用户可以按创建时间、更新时间排序项目列表
- **需求文档**：用户可以按创建时间排序项目列表
- **问题**：遗漏了"更新时间"排序
- **扣分**：-5分
- **影响**：需要补充一个排序选项，但不影响整体设计

**示例 2：优先级理解偏差**
- **原始需求**：系统必须支持导出为 Excel，可选支持导出为 PDF
- **需求文档**：系统支持导出为 Excel 和 PDF
- **问题**：未区分"必须"和"可选"
- **扣分**：-5分
- **影响**：可能导致开发优先级错误

**示例 3：范围理解偏差**
- **原始需求**：支持上传图片文件（JPG、PNG）
- **需求文档**：支持上传所有类型的文件
- **问题**：扩大了文件类型范围
- **扣分**：-5分
- **影响**：需要调整文件类型限制

---

**如何识别理解偏差**：

1. **逐句对照原始需求**：
   - 将需求文档的每一句与原始需求对照
   - 检查是否有曲解或遗漏

2. **关键词检查**：
   - 注意"必须"、"应该"、"可以"等关键词
   - 注意"所有"、"部分"、"仅"等范围词
   - 注意"和"、"或"等逻辑词

3. **场景验证**：
   - 用具体场景验证需求是否符合原始意图
   - 检查边界情况是否符合预期

4. **反向验证**：
   - 假设按此需求实现，是否能满足用户期望？
   - 是否会产生用户不期望的行为？

**检查要点**：
- [ ] 对原始需求的理解是否准确？
- [ ] 是否存在曲解用户意图的情况？
- [ ] 是否与原始需求文档一致？
- [ ] 是否正确理解了需求的优先级？
- [ ] 是否正确理解了需求的范围？

---

#### 2.2 表述清晰度（10分）

**评分项说明**：
- **存在歧义表述**：-3分/项
- **使用模糊词汇**（"快速"、"适当"、"尽可能"）：-2分/项

**详细说明**：

**1. 歧义表述（-3分/项）**

**什么是歧义表述**：
- 一句话可以有多种理解
- 不同人读到会有不同的解释
- 缺少必要的限定和说明

**常见歧义类型**：

**类型 1：范围不明确**
- ❌ 歧义：用户可以删除项目
- ✅ 清晰：用户可以删除自己创建的项目
- **问题**：未明确能删除哪些项目（自己的？所有的？）

**类型 2：条件不明确**
- ❌ 歧义：系统应该发送通知
- ✅ 清晰：当订单状态变更时，系统应该通过邮件发送通知给下单用户
- **问题**：未明确何时发送、通过什么方式发送、发送给谁

**类型 3：行为不明确**
- ❌ 歧义：系统应该处理失败的请求
- ✅ 清晰：当 API 请求失败时，系统应该重试 3 次，每次间隔 1 秒，3 次后仍失败则返回错误
- **问题**：未明确如何处理（重试？忽略？报错？）

**类型 4：数量不明确**
- ❌ 歧义：系统应该支持大量并发用户
- ✅ 清晰：系统应该支持 1000 个并发用户
- **问题**："大量"是多少？每个人理解不同

**类型 5：时间不明确**
- ❌ 歧义：系统应该快速响应
- ✅ 清晰：API 响应时间应该在 200ms 以内（P95）
- **问题**："快速"是多快？没有具体标准

**如何识别歧义**：
1. **多人理解测试**：让不同人读需求，看理解是否一致
2. **5W1H 检查**：Who（谁）、What（什么）、When（何时）、Where（何处）、Why（为何）、How（如何）
3. **边界测试**：考虑极端情况，看需求是否仍然清晰

**评分示例**：
- **每个歧义表述**：-3分
- **轻微歧义但可推断**：-1分

---

**2. 模糊词汇（-2分/项）**

**什么是模糊词汇**：
- 没有明确标准的形容词或副词
- 不同人有不同理解
- 无法量化或验证

**常见模糊词汇及替代方案**：

| 模糊词汇 | 问题 | 清晰替代 |
|---------|------|---------|
| 快速 | 多快算快？ | 响应时间 < 200ms |
| 适当 | 什么是适当？ | 具体的数值或范围 |
| 尽可能 | 尽到什么程度？ | 明确的目标值 |
| 大量 | 多少算大量？ | 具体的数量 |
| 经常 | 多频繁算经常？ | 具体的频率 |
| 简单 | 多简单算简单？ | 具体的操作步骤数 |
| 复杂 | 多复杂算复杂？ | 具体的复杂度指标 |
| 高性能 | 多高算高？ | 具体的性能指标 |
| 稳定 | 多稳定算稳定？ | 可用性 99.9% |
| 安全 | 多安全算安全？ | 具体的安全措施 |

**示例对比**：

**示例 1：性能需求**
- ❌ 模糊：系统应该快速响应用户请求
- ✅ 清晰：API 响应时间应该在 200ms 以内（P95），500ms 以内（P99）

**示例 2：容量需求**
- ❌ 模糊：系统应该支持大量用户
- ✅ 清晰：系统应该支持 10000 个注册用户，1000 个并发在线用户

**示例 3：可用性需求**
- ❌ 模糊：系统应该尽可能稳定
- ✅ 清晰：系统可用性应该达到 99.9%（每月停机时间不超过 43 分钟）

**示例 4：安全需求**
- ❌ 模糊：系统应该足够安全
- ✅ 清晰：系统应该支持 HTTPS 传输，密码使用 BCrypt 加密，支持 JWT Token 认证

**示例 5：易用性需求**
- ❌ 模糊：系统应该简单易用
- ✅ 清晰：用户完成注册流程不超过 3 步，每步操作不超过 5 个字段

**如何避免模糊词汇**：
1. **使用具体数值**：用数字代替形容词
2. **使用可测量的指标**：用可以验证的标准
3. **使用 EARS 语法**：强制使用明确的条件和响应
4. **参考行业标准**：使用行业公认的标准

**评分示例**：
- **每个模糊词汇**：-2分
- **模糊但可接受**（如"用户友好"）：-1分

---

**检查要点**：
- [ ] 需求描述是否清晰明确？
- [ ] 是否避免了模糊词汇？
- [ ] 是否易于理解，无歧义？
- [ ] 是否使用了具体的、可测量的标准？
- [ ] 不同人读到是否会有相同的理解？

---

### 3. 一致性（20分）

一致性评估需求文档内部是否自洽，以及是否与原始意图保持一致。不一致的需求会导致设计和实现的混乱，增加沟通成本。

#### 3.1 内部一致性（10分）

**评分项说明**：
- **需求之间存在冲突**：-5分/项
- **术语使用不一致**：-2分/项

**详细说明**：

**1. 需求之间存在冲突（-5分/项）**

**什么是需求冲突**：
- 两个或多个需求相互矛盾
- 无法同时满足多个需求
- 需求之间的逻辑关系不合理

**常见冲突类型**：

**类型 1：功能冲突**
- **示例**：
  - 需求 A：用户删除项目后，项目立即从系统中移除
  - 需求 B：用户可以恢复已删除的项目
- **冲突**：如果立即移除，就无法恢复
- **扣分**：-5分
- **解决**：明确是逻辑删除还是物理删除

**类型 2：权限冲突**
- **示例**：
  - 需求 A：普通用户只能查看自己创建的项目
  - 需求 B：所有用户都可以查看项目列表
- **冲突**：权限范围矛盾
- **扣分**：-5分
- **解决**：明确不同角色的权限范围

**类型 3：数据一致性冲突**
- **示例**：
  - 需求 A：订单创建后不可修改
  - 需求 B：用户可以修改订单地址
- **冲突**：是否可修改矛盾
- **扣分**：-5分
- **解决**：明确哪些字段可修改，哪些不可修改

**类型 4：流程冲突**
- **示例**：
  - 需求 A：订单支付成功后自动发货
  - 需求 B：订单需要人工审核后才能发货
- **冲突**：发货流程矛盾
- **扣分**：-5分
- **解决**：明确发货的前置条件

**类型 5：性能冲突**
- **示例**：
  - 需求 A：系统应该实时同步数据到备份系统
  - 需求 B：API 响应时间应该在 100ms 以内
- **冲突**：实时同步可能影响响应时间
- **扣分**：-5分
- **解决**：明确同步策略（异步同步）

**如何识别冲突**：
1. **交叉检查**：将所有需求两两对照，检查是否矛盾
2. **场景模拟**：用具体场景验证需求是否可以同时满足
3. **逻辑推理**：检查需求之间的逻辑关系是否合理
4. **状态机检查**：对于有状态的对象，检查状态转换是否合理

**评分示例**：
- **严重冲突（无法同时满足）**：-5分/项
- **轻微冲突（可以通过调整解决）**：-3分/项

**2. 术语使用不一致（-2分/项）**

**什么是术语不一致**：
- 同一概念使用不同的名称
- 不同概念使用相同的名称
- 术语定义前后不一致

**常见不一致类型**：

**类型 1：同一概念多个名称**
- **示例**：
  - 需求 A：用户可以创建"项目"
  - 需求 B：用户可以查看"工程"列表
  - 需求 C：用户可以删除"任务组"
- **问题**：项目、工程、任务组是同一个概念吗？
- **扣分**：-2分
- **解决**：统一使用"项目"

**类型 2：不同概念相同名称**
- **示例**：
  - 需求 A：用户可以创建"任务"（指项目中的工作项）
  - 需求 B：系统会自动创建"任务"（指后台定时任务）
- **问题**：两个"任务"是不同的概念
- **扣分**：-2分
- **解决**：区分为"工作项"和"定时任务"

**类型 3：术语定义不一致**
- **示例**：
  - 术语表：项目 = 一组相关的工作项
  - 需求描述：项目 = 一个独立的应用系统
- **问题**：定义前后不一致
- **扣分**：-2分
- **解决**：统一术语定义

**类型 4：缩写不一致**
- **示例**：
  - 需求 A：API 响应时间
  - 需求 B：接口响应时间
  - 需求 C：服务响应时间
- **问题**：API、接口、服务是同一个概念吗？
- **扣分**：-2分
- **解决**：统一使用"API"

**类型 5：大小写不一致**
- **示例**：
  - 需求 A：用户可以上传 PDF 文件
  - 需求 B：系统支持 pdf 格式
- **问题**：PDF 和 pdf 应该统一
- **扣分**：-1分（轻微）
- **解决**：统一使用大写"PDF"

**如何识别术语不一致**：
1. **术语表检查**：建立术语表，检查所有术语是否在表中
2. **全文搜索**：搜索相似的词汇，检查是否指同一概念
3. **上下文检查**：检查同一术语在不同上下文中的含义是否一致
4. **缩写检查**：检查缩写是否统一

**术语一致性最佳实践**：
1. **建立术语表**：在需求文档开头列出所有关键术语
2. **统一命名规则**：制定命名规则（如使用单数还是复数）
3. **避免同义词**：同一概念只用一个名称
4. **明确缩写**：首次使用时给出全称和缩写

**评分示例**：
- **每个术语不一致**：-2分
- **轻微不一致（如大小写）**：-1分

**检查要点**：
- [ ] 需求之间是否存在矛盾？
- [ ] 术语定义是否一致？
- [ ] 概念使用是否统一？
- [ ] 是否建立了术语表？
- [ ] 缩写和全称是否统一？

#### 3.2 与原始意图一致性（10分）

**评分项说明**：
- **偏离原始意图**：-5分/项
- **过度解读**：-3分/项

**详细说明**：

**1. 偏离原始意图（-5分/项）**

**什么是偏离原始意图**：
- 需求文档的内容与原始需求文档不一致
- 改变了原始需求的核心目标
- 遗漏了原始需求的关键约束

**常见偏离类型**：

**类型 1：改变核心目标**
- **原始意图**：开发一个内部使用的项目管理工具
- **需求文档**：开发一个面向公众的 SaaS 项目管理平台
- **问题**：改变了系统的定位和目标用户
- **扣分**：-5分
- **影响**：架构设计、安全要求、性能要求都会完全不同

**类型 2：遗漏关键约束**
- **原始意图**：系统必须在现有的 Oracle 数据库上运行
- **需求文档**：系统使用 MySQL 数据库
- **问题**：遗漏了技术约束
- **扣分**：-5分
- **影响**：技术选型错误，无法部署

**类型 3：改变业务规则**
- **原始意图**：订单金额超过 1000 元需要审批
- **需求文档**：所有订单都需要审批
- **问题**：改变了业务规则
- **扣分**：-5分
- **影响**：业务流程错误，影响用户体验

**类型 4：改变优先级**
- **原始意图**：第一阶段只实现基础的 CRUD 功能
- **需求文档**：第一阶段实现完整的工作流引擎
- **问题**：改变了开发优先级和范围
- **扣分**：-5分
- **影响**：开发周期和成本大幅增加

**如何识别偏离**：
1. **逐项对照**：将需求文档与原始需求文档逐项对照
2. **目标检查**：检查核心目标是否一致
3. **约束检查**：检查所有约束是否都被保留
4. **范围检查**：检查需求范围是否一致

**评分示例**：
- **严重偏离（改变核心目标）**：-5分/项
- **轻微偏离（细节调整）**：-3分/项

**2. 过度解读（-3分/项）**

**什么是过度解读**：
- 添加了原始需求中未提及的功能
- 这些功能不是必要的推导结果
- 增加了不必要的复杂度

**过度解读 vs 合理推导**：

**合理推导**（不扣分）：
- **原始需求**：用户可以登录系统
- **推导需求**：用户可以登出系统
- **理由**：登出是登录的必然配对功能

**过度解读**（扣分）：
- **原始需求**：用户可以登录系统
- **过度解读**：系统需要支持单点登录（SSO）、多因素认证（MFA）、社交账号登录
- **问题**：原始需求未提及这些高级功能

**常见过度解读类型**：

**类型 1：添加高级功能**
- **原始需求**：用户可以上传文件
- **过度解读**：
  - 系统需要支持文件版本管理
  - 系统需要支持文件协作编辑
  - 系统需要支持文件权限控制
  - 系统需要支持文件预览
- **问题**：原始需求只要求基本的上传功能
- **扣分**：-3分/项
- **影响**：大幅增加开发成本和复杂度

**类型 2：添加优化功能**
- **原始需求**：系统需要记录操作日志
- **过度解读**：
  - 系统需要支持日志分析和可视化
  - 系统需要支持日志告警
  - 系统需要支持日志导出
- **问题**：原始需求只要求记录日志，未要求分析功能
- **扣分**：-3分/项

**类型 3：添加扩展功能**
- **原始需求**：系统支持邮件通知
- **过度解读**：
  - 系统需要支持短信通知
  - 系统需要支持微信通知
  - 系统需要支持 App 推送
- **问题**：原始需求只提到邮件，未提及其他通知方式
- **扣分**：-3分/项

**类型 4：添加性能优化**
- **原始需求**：系统需要支持 100 个并发用户
- **过度解读**：
  - 系统需要支持水平扩展
  - 系统需要支持缓存
  - 系统需要支持读写分离
- **问题**：100 个并发用户不需要这些复杂的优化
- **扣分**：-3分/项

**如何判断是否过度解读**：
1. **必要性检查**：这个功能是否是满足原始需求的必要条件？
2. **明确性检查**：原始需求是否明确提到或暗示了这个功能？
3. **复杂度检查**：添加这个功能是否显著增加了系统复杂度？
4. **成本检查**：添加这个功能是否显著增加了开发成本？

**何时应该补充需求**：
如果发现需要添加原始需求未提及的功能，应该：
1. **与用户确认**：询问用户是否需要这个功能
2. **补充原始需求**：如果需要，先补充到原始需求文档
3. **记录理由**：说明为什么需要添加这个功能

**评分示例**：
- **每个过度解读的功能**：-3分
- **合理的功能推导**：不扣分

**检查要点**：
- [ ] 是否忠实于原始需求？
- [ ] 是否存在过度设计的倾向？
- [ ] 是否添加了不必要的需求？
- [ ] 添加的功能是否都与用户确认过？
- [ ] 是否保持了原始需求的核心目标和约束？

### 4. 可验证性（15分）

可验证性评估需求是否可以被测试和验证。不可验证的需求无法判断是否被正确实现，会导致验收困难。

#### 4.1 EARS 语法规范（8分）

**评分项说明**：
- **未使用 EARS 语法**：-2分/项
- **EARS 语法使用不当**：-1分/项

**详细说明**：

**什么是 EARS 语法**：
EARS（Easy Approach to Requirements Syntax）是一种结构化的需求编写语法，通过标准模式确保需求的清晰性和可验证性。

**EARS 的五种基本模式**：

**1. Ubiquitous（普遍需求）**
- **格式**：THE [system] SHALL [action]
- **用途**：描述系统必须始终满足的需求
- **示例**：
  - ✅ THE System SHALL 使用 HTTPS 加密所有数据传输
  - ✅ THE System SHALL 记录所有用户操作日志
  - ❌ 系统应该安全（未使用 EARS 语法，-2分）

**2. Event-driven（事件驱动需求）**
- **格式**：WHEN [trigger] THE [system] SHALL [action]
- **用途**：描述特定事件发生时系统的响应
- **示例**：
  - ✅ WHEN 用户点击登录按钮 THEN THE System SHALL 验证用户名和密码
  - ✅ WHEN 订单支付成功 THEN THE System SHALL 发送确认邮件给用户
  - ❌ 用户登录时验证密码（未使用 EARS 语法，-2分）

**3. State-driven（状态驱动需求）**
- **格式**：WHILE [state] THE [system] SHALL [action]
- **用途**：描述系统在特定状态下的行为
- **示例**：
  - ✅ WHILE 用户已登录 THE System SHALL 显示用户菜单
  - ✅ WHILE 订单处于待支付状态 THE System SHALL 允许用户修改订单
  - ❌ 登录后显示菜单（未使用 EARS 语法，-2分）

**4. Optional（可选需求）**
- **格式**：WHERE [feature] THE [system] SHALL [action]
- **用途**：描述可选功能或特定配置下的需求
- **示例**：
  - ✅ WHERE 启用多因素认证 THE System SHALL 要求用户输入验证码
  - ✅ WHERE 用户是管理员 THE System SHALL 显示管理菜单
  - ❌ 如果是管理员显示管理菜单（未使用 EARS 语法，-2分）

**5. Unwanted behavior（不期望行为）**
- **格式**：IF [condition] THEN THE [system] SHALL [action]
- **用途**：描述错误或异常情况下的处理
- **示例**：
  - ✅ IF 用户密码错误 THEN THE System SHALL 显示错误提示并拒绝登录
  - ✅ IF API 请求超时 THEN THE System SHALL 重试 3 次
  - ❌ 密码错误时显示提示（未使用 EARS 语法，-2分）

**EARS 语法使用不当的常见问题**：

**问题 1：混用模式**
- ❌ 错误：WHEN 用户已登录 THE System SHALL 显示菜单
- ✅ 正确：WHILE 用户已登录 THE System SHALL 显示菜单
- **说明**："已登录"是状态，应该用 WHILE 而非 WHEN
- **扣分**：-1分

**问题 2：缺少关键部分**
- ❌ 错误：WHEN 用户点击按钮 SHALL 提交表单
- ✅ 正确：WHEN 用户点击提交按钮 THEN THE System SHALL 提交表单
- **说明**：缺少 "THE System"
- **扣分**：-1分

**问题 3：条件不明确**
- ❌ 错误：WHEN 用户操作 THE System SHALL 响应
- ✅ 正确：WHEN 用户点击保存按钮 THEN THE System SHALL 保存数据并显示成功提示
- **说明**：条件和响应都不够具体
- **扣分**：-1分

**检查要点**：
- [ ] 是否使用了标准的 EARS 模式？
- [ ] EARS 关键字（THE、SHALL、WHEN、WHILE、IF、THEN、WHERE）使用是否正确？
- [ ] 语法结构是否规范？
- [ ] 条件和响应是否明确？

## 使用指南

### 评分流程

1. **准备阶段**
   - 确保需求文档已完成自检
   - 准备原始需求文档作为对照

2. **评分阶段**
   - 按照五个维度逐项检查
   - 记录发现的问题和扣分项
   - 计算各维度得分和总分

3. **反馈阶段**
   - 填写评分表
   - 列出主要问题和改进建议
   - 决定是否可进入下一阶段

4. **改进阶段**
   - 根据反馈进行改进
   - 重新评分验证

### 注意事项

- **客观公正**：基于事实评分，避免主观臆断
- **记录详细**：清楚记录每个扣分项的具体原因
- **建设性反馈**：不仅指出问题，还要提供改进建议
- **持续改进**：定期回顾评分标准，根据实践经验优化

## 常见问题

### Q1：如何判断是"严重理解偏差"还是"轻微理解偏差"？
**A**：严重理解偏差指完全曲解了需求意图，导致后续设计方向错误；轻微理解偏差指部分理解不准确，但不影响整体方向。

### Q2：什么情况下算"过度解读"？
**A**：添加了原始需求中未提及的功能，且这些功能不是必要的推导结果。

### Q3：如何判断验收标准是否"可测试"？
**A**：可测试的验收标准应该能够通过具体的操作步骤进行验证，并有明确的预期结果。

### Q4：评分后发现问题，是否必须立即改进？
**A**：根据评级决定：
- 不合格（<60分）：必须立即改进
- 需改进（60-69分）：必须改进后才能进入下一阶段
- 合格（70-79分）：可以进入下一阶段，但需记录改进点
- 良好及以上（≥80分）：可直接进入下一阶段

## 验收标准明确性详细说明

### 缺少验收标准（-3分/项）

**为什么需要验收标准**：
- 明确"完成"的定义
- 提供测试依据
- 避免理解歧义
- 支持验收决策

**每个需求都应该有验收标准**：
- ❌ 错误：WHEN 用户点击登录按钮 THEN THE System SHALL 验证用户身份
  - **问题**：没有验收标准，无法判断如何验证是否正确实现
  - **扣分**：-3分

- ✅ 正确：WHEN 用户点击登录按钮 THEN THE System SHALL 验证用户身份
  - **验收标准**：
    1. 用户名和密码正确时，跳转到首页并显示用户信息
    2. 用户名或密码错误时，显示"用户名或密码错误"提示
    3. 用户名为空时，显示"请输入用户名"提示
    4. 密码为空时，显示"请输入密码"提示
    5. 连续 5 次密码错误后，账号锁定 30 分钟

### 验收标准不可测试（-2分/项）

**可测试验收标准的特征**：
1. **具体的输入**：✅ 用户输入用户名"admin"和密码"123456"
2. **明确的操作**：✅ 用户点击"登录"按钮
3. **可观察的输出**：✅ 页面跳转到首页，顶部显示"欢迎，admin"
4. **可测量的指标**：✅ API 响应时间 < 200ms

**验收标准模板**：
```
GIVEN [前置条件]
WHEN [操作]
THEN [预期结果]
```
