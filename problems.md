# 分析需求遇到的问题
- 需求不完善
    - 整理一个原始的需求，写入到一个文件，不需要结构化，让Kiro将其结构化，基于这个原始版本来生成需求，在需求分析阶段，发现有需要补充的，补充到原始的需求文档，这样可以再次生成requirement。
- 无法判断需求是否完善
    - 让Kiro来判断是否完善，要求必须与原始需求一致，需求内部不冲突，需求可验证
- 每次验证后，结果不一致
    - 多验证几次，发现有用的信息，就记录到原始需求，会慢慢减少随机性，趋于稳定
- 过于主观或者过于技术化
    - 请移除过于主观或者过于技术化的需求
    - 将"所有"、"清晰"等主观词汇改为具体的范围和标准
- 需求的验证标准不够明确
    - 让Kiro分析哪些需求的验收标准不够明确，占比是多少，控制在10%一下，不涉及核心需求即可
    - 让Kiro统计验证方法的分步，尽可能多的使用运行时验证，其次是编译验证，最后是静态验证

# 设计时遇到的问题
- 过度设计
    - 让Kiro检查哪些是过度设计的，给出这样设计的考量，如果是需要的，反补需求，可有可无的，移除
- 过于细节，出现了大量的编码
    - 明确告诉Kiro，不要编码，明确要做哪些设计，例如：架构设计，模块设计，时序设计等等
- 设计阶段仍旧有很多不够明确的
    - 在设计之前，务必让Kiro就形成的需求分析做澄清，明确哪些是需要补充和确认的。例如：集成mybatis plus时，有大量的配置要澄清，mapper的包路径，使用什么连接池，Druid 还是HikariCP，操作人是上下文传递还是通过参数传递。。。

# 任务分解中遇到的问题
- 任务没有覆盖所有的需求
    - 优先保证任务能覆盖所有的需求，再保证每个任务是可执行，可验证，特别是验证标准要明确
- 验证任务时，指定了过于具体的操作步骤
    - 只验证结果，指定条件，验证预期，不需要指定操作过程
